## Architectural defenses / guardrails review

Verify that the implementation preserves the project's existing architectural
boundaries. Check only defenses stated in AGENTS.md, docs/references, the current
plan, or established code patterns. Do not invent rules or expand branch scope.

Specifically verify:
- FastAPI endpoints remain thin and do not absorb business logic;
- request/response boundary schemas own transport adaptation when the plan or
  project rule requires it;
- use cases receive domain-level inputs, not raw FastAPI requests, TestClient
  responses, or persistence models;
- routers are registered through `src/api/routers.py::root_router`;
- endpoint DI uses `DishkaRoute` and `FromDishka[...]` where dependencies are managed;
- auth, validation, exception mapping, and middleware paths are not bypassed;
- persistence implementations live under `src/storages`, not `repositories`;
- transactions remain provider-owned when a DB layer exists;
- tests use centralized fixtures/helpers instead of local app/client/container setup;
- no unrelated rule is changed to make the task easier.

Report violations with severity:
- critical/major if behavior, validation, security, permissions, exception
  handling, persistence safety, or a declared architecture boundary is broken;
- minor if only documentation/test clarity is missing;
- false positive if the suggested simplification conflicts with explicit project rules.

## Counter / metrics persistence-safety review

For any branch that adds or changes counters, metrics, denormalized counters, or
database-side updates, verify persisted-value safety:
- updates are atomic or otherwise safe for the configured persistence layer;
- same-response values are not computed from stale in-memory objects;
- returned objects are refreshed or synchronized when needed;
- tests cover persisted and response values separately when both matter;
- `HEAD`, `OPTIONS`, failed lookups, validation errors, and auth failures do not
  mutate counters unless explicitly required;
- write logic stays in the declared write boundary.

Severity:
- critical/major if stale values, lost updates, or non-idempotent non-mutating
  requests can affect user-visible data;
- minor if only documentation or test clarity is missing;
- false positive if the branch does not mutate counters/metrics.
