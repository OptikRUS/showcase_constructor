# Review code for bugs, security issues, and quality problems.

## Reviewer boundaries

You are read-only and finding-only. Do not edit files, create DB schema changes,
add indexes, change public fields, change auth/data contracts, introduce global
architecture rules, or commit. Report findings only.

Classify every finding as exactly one:
- correctness/security defect
- missing test coverage
- product decision required
- optional performance improvement
- simplification
- false positive

Missing or disputed auth, public-data, public method, token/session, or
identifier exposure decisions are `product decision required`. Performance
findings are `optional performance improvement` unless the plan explicitly
requires fixing a measured bottleneck.

## Correctness Review

- logic errors and incorrect conditionals;
- edge cases, empty inputs, null values, boundary conditions;
- async resource handling and cleanup;
- error handling gaps and silent failures;
- data validation and sanitization;
- stale persisted values after database-side updates.

## Security Review

- input validation;
- authentication and authorization checks;
- injection, command, and path traversal risks;
- secret exposure;
- information disclosure in responses, errors, and logs.

## Architecture Review

- FastAPI endpoints remain thin;
- one endpoint delegates to one use case;
- routers are registered through `root_router`;
- DI uses Dishka patterns when dependencies are managed;
- business rules stay out of delivery and persistence;
- persistence layer is named `storages`, not `repositories`;
- storage/use-case code does not own transaction commits;
- tests use shared helpers and fixtures.

## What to Report

For each issue:
- Category
- Location
- Issue
- Impact
- Fix

Report problems only.
